# Gotchas — UI (Unity / TextMeshPro / layout)

Load when: editing `BubbleBuffer.cs` UI code, `UIHelpers.cs`, or adding/modifying any Unity layout.

## Bullet Gotchas

- **`sourceControlObj` visibility**: Controlled by `sourceCount > 1 || hasSpellProviders` in `UpdateDetailsView`. Hidden entirely for songs (sourceCount=0, no spell providers) and equipment-only buffs. Any UI control that must be visible for ALL buff types (e.g., combat start checkbox) must be parented outside `sourceControlObj` — use `spellInfoSection` with `ignoreLayout = true` instead.
- **`source-controls-section` needs minHeight ≥ 110**: The section hosts two vertical stacks (`prioSide`: prioLabel + extendRod + combatStart + roundLimit; `toggleSide`: 4 source toggles). With minHeight=30 the containers were only ~56 high, and togglePrefab's fixed sizeDelta pushed the bottom children (Use Equipment, Combat Start) outside the parent bounds — clicks silently failed. Keep `sourceControlsSection` at 110+ so all children fit vertically.
- **`TextMeshProUGUI` competes with `LayoutElement` at default priority**: TMP implements ILayoutElement (priority 1) and can make a LayoutElement's `preferredHeight` be ignored, collapsing the text to 0 height. For text with a fixed target height (e.g. the priority label), set `LayoutElement.layoutPriority = 2` AND `minHeight` to force the LayoutElement to win. Also disable `enableWordWrapping` when the text should stay on one line.
- **Caster portrait index ≠ CasterQueue index**: `BufferView.casterPortraitMap` maps portrait indices to CasterQueue indices after deduplication. Always use the map when translating portrait clicks to CasterQueue entries.
- **Debug Unity UI positions with `GetWorldCorners()`**: When UI elements appear mispositioned, add `Main.Log` calls with `RectTransform.GetWorldCorners()` to compare actual pixel positions at runtime. Anchor math from code is error-prone — verify empirically via Player.log. For "why is this child outside its parent" bugs, walk the parent chain with `transform.parent` and log each level's worldW/worldH + anchorMin/Max/sizeDelta — reveals which ancestor clamps the size.
- **Don't destroy UI elements during their click callbacks**: `GameObject.Destroy` (deferred) and `DestroyImmediate` both corrupt Unity's EventSystem mid-click. Let `ShowBuffWindow` handle rebuilds via its size-mismatch detection instead.
- **`OwlcatButton` via `AddComponent` doesn't render**: Needs internal layer structure. Use `MakeButton()` with `buttonPrefab` (static field set in `CreateWindow`) for game-styled buttons.
- **ScrollRect needs raycast target for wheel events**: Add transparent `Image` with `raycastTarget = true` to the Viewport. Without it, only drag-scroll works.
- **`GlobalBubbleBuffer.Buttons` references go stale after save/load**: The `Buttons` list contains OwlcatButton references that become destroyed Unity objects when the UI is reinstalled. Always null-guard individual buttons in `ForEach` lambdas. In EventBus handlers, separate UI operations from game logic in distinct try-catch blocks so a stale button doesn't block other functionality.
- **Cloned action-bar prefabs carry invisible `ContentSizeFitter` + `CanvasGroup`**: `m_DragSlot.m_ConvertedView.gameObject` (used as the Settings popup frame in `MakeSettings`) instantiates with CSF whose `verticalFit=PreferredSize`. When a child layout contributes an ILayoutElement (original code had `GridLayoutGroup` on panel), CSF sizes panel to fit all cells — the "overflow" was cells extending horizontally past the narrow 100px `sizeDelta`. When restructuring (e.g. moving the GLG onto a ScrollRect Content child), panel's direct child becomes something with no preferred-size contribution → CSF collapses panel to 0×0 → only a ~1px frame renders. Fix: add `panel.DestroyComponents<ContentSizeFitter>()` alongside the existing `ActionBarConvertedPCView`/`GridLayoutGroup` destroys when refactoring.
- **`Mask` + `Image.color.alpha == 0` silently hides all masked children**: The documented "ScrollRect needs raycast target: transparent Image with raycastTarget=true" recipe works for wheel-event raycasting, but **combining that transparent Image with a `Mask` component breaks masking** — the alpha-0 Graphic may not emit a stencil write, so children reference an empty stencil buffer and get discarded. Use `RectMask2D` instead (rect-based clipping, no Graphic dependency). Keep the transparent Image for raycast alongside `RectMask2D`.
- **`BuffProvider.spell` (`who.spell`) is null for song/activatable providers**: The caster popout in `UpdateDetailsView` dereferences `who.spell` for the arcanist-only toggles (`who.spell.IsArcanistSpell`, `who.spell.Spellbook`, `who.spell.Blueprint.School`). For songs/activatables `who.spell == null` → NPE that kills the whole details view the moment a song performer portrait is expanded, making Ban/Cap unreachable. Guard the spell-specific toggles behind `who.spell != null` (Ban + Cap need no spell). Distinct from `BubbleBuff.Spell` null (buff-level) — this is the per-caster provider's spell.
- **UMM hot-reload swaps the assembly but does NOT destroy existing UI GameObjects**: `[EnableReloading]` (DEBUG only) plus `GlobalBubbleBuffer.Uninstall()` unsubscribes EventBus handlers but leaves the buff-window/settings-panel GameObjects in the scene. Their `OnLeftClick` delegates are still bound to old-assembly method pointers. After Shift+B, a new DLL is loaded but clicking the gear button runs the OLD click handler. Verification pattern: if `grep '\[SETT-DBG\]' Player.log | tail` doesn't show lines added in the new build, the new assembly is loaded but the UI wasn't rebuilt — full game restart is the only guaranteed refresh, or force `TryInstallUI()` (Shift+I) + ensure the window is destroyed first.

## Unity UI Layout Patterns


- **`MakeButton()` breaks layout groups**: Sets point-anchors `(0.5, 0.5)` → zero size in HLG/VLG. Always reset anchors to stretch `(0,0)→(1,1)` after calling `MakeButton()`.
- **`childControlHeight=true` + `childForceExpandHeight=false`**: Correct combo for LayoutElement-driven sizing. `childControl=false` ignores LayoutElement entirely (children collapse to RectTransform default = 0). `childForceExpand=true` stretches beyond preferred size.
- **`buttonPrefab` is designed for full-width text buttons**: Has internal Image/decoration layers that look broken at small sizes (<60px). Don't use as icon buttons.
- **`buttonPrefab` minimum height ~38px**: Below this threshold, internal decoration layers become invisible/transparent. Always set `preferredHeight >= 38` on rows containing buttonPrefab instances.
- **`layoutPriority` on LayoutElement**: Higher priority means the parent LayoutGroup uses THIS element's preferred/flexible values instead of calculating from children. Use `layoutPriority = 3` on row LayoutElements to override buttonPrefab's internal preferred sizes.
- **Anchor-based children inside VLG sections**: VLG with `childControlWidth/Height = true` controls the section's RectTransform. Anchor-based grandchildren position relative to the section rect — this works correctly. But inner LayoutGroups can fight with anchors; prefer one approach per container.
- **`UIHelpers.Create()` / `AddTo()`**: Uses `SetParent(parent)` without `false` — can cause positioning bugs. Use `SetParent(parent, false)` when positioning matters.
- **Scrolling a cloned game-prefab container: prefer "wrap above" over "restructure inside"**. Adding a ScrollRect inside the container and moving its existing children into a new Viewport/Content hierarchy risks disturbing inherited layout components (`ContentSizeFitter`, `LayoutElement`, `CanvasGroup` on game prefabs — see Bullet Gotchas above). Safer pattern: leave the existing container alone, create a new parent `GameObject` with `ScrollRect` + `Viewport` (`RectMask2D`) and re-parent the whole existing container as the ScrollRect's Content. Unchanged inner layout = zero cascade risk from unknown inherited components. The v1.13.6 settings-popup refactor hit three separate bugs (CSF collapse, Mask+alpha-0 stencil fail, UMM hot-reload ghost UI) that all would have been avoided by this pattern.

- **Caster popout binds via raw `SelectedCaster` index — always go through `TryGetSelectedProvider`**: `SelectedCaster` is a plain CasterQueue index that can go stale (queue rebuild/shrink while popout open) or be re-targeted (`SortProviders` reorders in place — close the popout there). Since v1.14.5 the popout has a provider-selector row cycling through ALL of a unit's providers (multiclass spellbooks, scrolls), so non-first indices are routine. New popout listeners must use `TryGetSelectedProvider(out buff, out caster)` and value-diff guards (programmatic `isOn =` rebinds fire `onValueChanged` → unguarded `Recalculate(true)` cascades on every cycle-arrow click).
- **`currentSelectedSpell` is re-matched by Key in `view.OnUpdate`**: after `RecalculateAvailableBuffs` the BuffList holds NEW BubbleBuff instances; without the re-match, popout mutations land on an orphaned instance and the trailing `Save()` serializes the unmutated rebuilt one (silent data loss). Any new code path that rebuilds BuffList outside `Recalculate` must also re-match or null the selection.

- **`CreateWindow` re-runs on party-size change while the controller and `view` persist — every Subscribe on a persistent ReactiveProperty must go into `windowSubscriptions`**: the list is disposed at the top of `CreateWindow`; `BindToView` returns its `IDisposable` for exactly this. Untracked subscriptions stack per rebuild: `RefreshFiltering` fires N times per toggle, `SortByName` saves N times, stale `currentSelectedSpell` handlers touch destroyed widgets. Same rebuild trap: `_ToAnimate` must be cleared before re-adding (`AssetLoader.Materials` returns shared Material instances), and `GlobalBubbleBuffer.Buttons` accumulates destroyed buttons across UI reinstalls — iterate with a null guard.
