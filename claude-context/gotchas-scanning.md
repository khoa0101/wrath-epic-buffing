# Gotchas — Scanning (BufferState / item & activatable discovery)

Load when: editing `BufferState.cs` (scan/discovery logic), adding a new item source, adding a new activatable-ability source, or touching pet/companion iteration.

## Bullet Gotchas

- **`BuffProvider.CanTarget` crashes when `spell` is null**: `AddActivatable()` sets `spell = null` for ALL activatables (songs, class abilities, AND item-backed). `CanTarget` and `SelfCastOnly` now guard `spell == null` as self-cast-only. Any new code accessing `provider.spell` must null-check first.
- **`AddActivatable()` bypasses `GetBeneficialBuffs`**: Unlike `AddBuff()`, activatables are added directly without checking for beneficial effects. Item-backed activatables are filtered by `blueprint.Buff != null`, but some items with buffs (e.g. summoning figurines) still pass — their buff triggers a summon rather than granting stats. Currently accepted as minor noise in the Equipment tab.
- **Magic Deceiver fused spells (MagicHack system)**: Fused spells use template `BlueprintAbility` shells (`MagicHackDefaultSlot1-10`, `MagicHackTouchSlot1-10`) with empty actions/icons. Actual data lives in `AbilityData.MagicHackData` (`Spell1`, `Spell2`, `Name`, `DeliverBlueprint`, `SpellLevel`). The game engine routes `Name`, `Icon`, `CanTargetAlly`, `CanTargetEnemies`, `TargetAnchor` through `GetDeliverBlueprint()` which returns `MagicHackData.DeliverBlueprint` for fused spells. The `MagicDeceiverSpellbook` is `Spontaneous: true, IsArcanist: false, AllSpellsKnown: true` with its own curated spell list (mostly offensive — no standard buffs like Haste/Shield). Fused spells are stored via `Spellbook.AddCustomSpell()`.
- **Archetype spellbooks can override `IsArcanist`**: `MagicDeceiverSpellbook` sets `IsArcanist: false` despite being an Arcanist archetype. Don't assume Arcanist archetypes use the Arcanist scanning branch — always check the spellbook blueprint's flags.
- **Weapon enchantment spells use `EnhanceWeapon` action**: Magic Weapon, Keen Edge, Magic Fang and similar spells all use `Conditional` → `EnhanceWeapon` + `ContextActionApplyBuff`. Handled by `EnhanceWeaponEffect` in `IBeneficialEffect.cs`. If a new weapon enchantment spell is reported missing, check for this pattern first.
- **Enchant-pool abilities (Arcane Weapon, Sacred Weapon, Weapon Bond) — `DefaultEnchantments` is NOT a reliable presence marker**: `ContextActionWeaponEnchantPool.RunAction` (IL-verified) first applies the property enchants toggled via Choice activatables (`AddBondProperty` buff components), then `DefaultEnchantments[bonus−1]` only when `bonus = min(max(0, 5 − weapon's own enhancement), remaining pool)` is positive — a +5 weapon or a pool fully spent on properties gets NO default enchant on the weapon, so an `IsPresent` matching only DefaultEnchantments GUIDs never fires and the routine re-casts every run (zh Nexus report). Since the fix, `WeaponEnchantPoolEffect` also records `action.EnchantPool` and `IsPresent` consults `UnitPartEnchantPoolData.m_EnchantPoolDataDescriptions`: a description with a matching pool type whose booked enchant-fact IDs still resolve via `item.Facts.FindById` means the effect is live (the game removes those facts on unequip — `HandleEquipmentSlotUpdated` — and on duration expiry, so stale IDs correctly read as absent). `ContextActionShieldWeaponEnchantPool` stays unscanned: the main-hand action from the same cast suffices as marker.
- **Self-gated shared buffs — a flattened `AppliedBuffs` set is an unreliable presence marker for conditional "re-grant" abilities**: `GetBeneficialBuffs` flattens ALL `Conditional` branches into one `AppliedBuffs` set, losing which branch actually fires. The Arcanist exploit `ArcanistExploitArmoredMaskAbility` (buff `5274ddc2…`) re-applies the vanilla **MageArmorBuff** (`a92acdf1…`) / MageArmorBuffMythic (`355be068…`) when the target lacks armor, else applies its own bonus buff — so its `AppliedBuffs` = {MageArmor, MageArmorMythic, bonus}. With Mage Armor already up, `IsPresent` overlapped on the shared MageArmorBuff and skipped the exploit even though its bonus was absent (Nexus report). Fix: `BlueprintAbility.GetAbsenceCheckedFacts()` collects every fact checked via `ContextConditionHasFact(Not:true)` in the action tree; `AbilityCombinedEffects` intersects that with `AppliedBuffs` → `SelfGatedBuffs` (buffs the ability both applies AND guards on its own absence = "re-grant if missing"). `IsPresent` drops `SelfGatedBuffs` from the marker set, with an **empty-fallback**: if excluding them empties the markers (a pure "apply X if absent" buff whose only payload is X), keep them so it isn't re-cast every run. Mage Armor's own spell applies the same buffs WITHOUT a `HasFact(Not)` guard, so its markers are untouched. Edge case (accepted): Armored Mask as a character's ONLY armor builds up over two routine runs (run 1 → mage armor, run 2 → bonus on top). Whenever a "buff skipped when a different spell sharing its buff is active" report shows up, check for this shared-buff / self-gating pattern.
- **`Game.Instance.Player.RemoteCompanions`** is the correct API for benched-but-available companions. `AllCharacters` includes dead, ex-companions, and summons. `RemoteCompanions` includes pets — filter with `unit.Get<UnitPartPet>() != null`. It's `IEnumerable`, not `ICollection` — use `.Count()` (LINQ), not `.Count` property (CS0428 "cannot convert method group to int").
- **`IsInGame` semantics**: `False` = benched/remote companion (not in scene), `True` = in scene (active party + summons). Counterintuitive — benched companions are NOT "in game".
- **`spell.CanTarget()` rejects out-of-scene units**: The game's ability targeting system requires targets to be in the current scene. For config-only targeting (reserve companions), bypass this check.
- **`ActivatableAbility.SourceItem` distinguishes source**: Inherited from `Fact`. Null = class feature (Battle Spirit, Judgments, Rage). Non-null = item-granted (metamagic rods, quivers). Critical filter for the activatable scan.
- **`ActivatableAbilityResourceLogic` = has resource cost**: Component on `BlueprintActivatableAbility`. Presence means the ability consumes resources (rounds/day, charges). Absence means free toggle (Power Attack, Wings, Combat Styles). Filter out free toggles in scanning or the Ability tab drowns in noise.
- **Item-backed activatables bypass the QuickSlot scan**: `BlueprintItemEquipmentUsable` items with `Ability == null` (metamagic rods, ammunition quivers) grant their effect via ActivatableAbility instead of BlueprintAbility. They fall through the QuickSlot scan (skipped because Ability is null) and must be routed via the `ActivatableAbility.SourceItem != null` branch to `AddActivatable(..., Category.Equipment, sourceItem)`.
- **`BlueprintBuff.FxOnStart` fallback in `IsBeneficial`**: Visual-only buffs like MageLightBuff have only `UniqueBuff` (infrastructure) but their actual effect is the `FxOnStart` light asset. `IsBeneficial` must accept buffs with a non-null `FxOnStart.AssetId` even when all components are infrastructure, otherwise Light cantrip and similar get rejected.
- **`AddBuff()` has two separate filter concerns**: `skipDamageFilter` (relaxes `GetBeneficialBuffs` for abilities with damage components) should apply to ALL `Category.Ability` entries. The self-target fallback (accepts abilities with no detected effects) should only apply to `isClassAbility` (`sourceItem == null`). Don't conflate the two — item-backed abilities need the relaxed damage filter but must NOT hit the self-target fallback.
- **`SpellsWithBeneficialBuffs` cache key is GUID-only**: The cache doesn't include `skipDamageFilter` as part of the key. Currently safe because class abilities and item-backed abilities have disjoint GUIDs, but if that ever changes the first scan wins and later scans get a stale result.
- **`Category.Toggle` = free-toggle activatables without `ActivatableAbilityResourceLogic`** (Power Attack, Shifter Claws, Monk stances, Arcane Strike). Routed via the `else` branch after `hasResourceLogic` in `BufferState.cs` activatable scan. Tab is **always visible** (no `TogglesEnabled` setting — per-ability opt-in already happens via the character portrait grid inside the tab). Round-limit slider hidden via `buff.Category != Category.Toggle` guard in `UpdateDetailsView`. Execution reuses the existing `ExecuteCombatStart()` Phase 0 unchanged (all activatables flow through the same `IsActivatable` filter).
- **Some scanned activatables are targeted (`Blueprint.IsTargeted`) and can't be completed by `IsOn=true`**: the no-resource Toggle scan surfaces e.g. the stock Mount toggle. Since v1.16.0 Phase 0 special-cases these via `BuffExecutor.TryMountTargeted` (resolves the rider's single suitable pet, calls `UnitPartRider.Mount` directly — see `gotchas-casting.md`). Scan-side nothing changes: they're surfaced like any free toggle; the executor decides.
- **Scan category summary log** (`BufferState.cs`, ~line 425): `Main.Log($"Scan complete: {list.Count} buffs (Buff=..., Ability=..., Equipment=..., Song=..., Toggle=...)")`. When adding a new `Category` enum value, extend BOTH the switch counters AND the log format string, otherwise the new category silently vanishes from diagnostics.
- **Two distinct ConversionsProvider patterns — distinguish via `ActivationDisable`**: `IActivatableAbilityConversionsProvider` is used for two very different mechanics. Pattern A (Shifter's Fury): parent blueprint has `ActivationDisable` component → `CanTurnOn()` pinned to false → `IsOn=true` is a no-op. The parent is a menu stub; real activators live on `ShiftersFuryPart.AppliedFacts` (one per wielded natural weapon). Conversions share the parent's DisplayName so must be deduplicated. Pattern B (Chimeric Aspect / Greater / Final): parent has `ActivatableAbilityResourceLogic` (real resource-costing Activatable), NO `ActivationDisable`. Conversions (Bear/Boar/…) are sub-form selections with distinct DisplayNames — BOTH parent AND children belong in the UI, do NOT deduplicate. Scan filter must be `blueprint.GetComponent<ActivationDisable>() != null && !(ConversionsProvider is ShiftersFury)`, not `ConversionsProvider != null` (that would kill Pattern B — v1.13.1 regression cost 2 more releases). See `BufferState.cs` activatable scan.
- **Shifter's Fury activation dispatch**: `BuffExecutor.ResolveActivationTarget()` routes `ShiftersFury` parents to `ShiftersFuryPart.AppliedFacts[State.SelectedWeaponIndex]` (fallback index 0). `IsEffectivelyOn()` checks `ShiftersFuryPart.m_IsOn` because the parent's `IsOn` is pinned to false by `ActivationDisable`.
- **Don't filter activatables on `IsRuntimeOnly || HiddenInUI` alone**: Mutual-exclusive class features (Chimera Aspect / Greater Chimera Aspect forms, Fiendflesh aspects, Weretouched forms) set both flags but are legitimate user-togglable abilities — blanket-filtering kills them (regression in v1.13.1). Dedup variant-conversions via `ConversionsProvider.GetConversions()` per unit: build a `HashSet<BlueprintGuid>` of every conversion's blueprint guid, then skip any activatable whose guid is in that set. See `BufferState.cs` scan loop.
- **Equipment-activatable charges gate must respect `SpendCharges`**: `srcItem.Charges <= 0` alone falsely skips permanent-toggle items like Crimson Banner (`SpendCharges: false`) — the nominal Charges value is never consumed or refreshed. Gate with `&& (srcItem.Blueprint as BlueprintItemEquipmentUsable)?.SpendCharges != false` so only real charge-consuming items get skipped when empty. See `BufferState.cs` equipment-activatable branch.
- **`GetBeneficialBuffs` must recurse through `ContextActionRepeatedActions`**: Like `Conditional`, `ContextActionRandomize`, `ContextActionPartyMembers`, etc., this wrapper holds an `ActionList` (`repeat.Actions.Actions`) of nested actions. DLC5 spells like Vengeful Comets stack their `ContextActionApplyBuff(ToCaster: true)` inside a `ContextActionRepeatedActions` body — without the recursive case the inner ApplyBuff is never seen, the spell registers as "no applied effects", and `AddBuff` rejects it (the self-target fallback only applies to `isClassAbility && TargetAnchor.Owner`, not to spellbook spells). Whenever a new wrapper-style action shows up in user-reported missing buffs, check `ExtentionMethods.GetBeneficialBuffs` first — every action whose payload is an `ActionList` must have a recurse branch.
- **`AddActivatable` must register a per-character provider, not blueprint-dedupe**: `BuffsByKey` is keyed on the activatable's blueprint GUID, so when the same toggle/ability lives on multiple party members (Power Attack, Combat Style, Inspire Courage on two bards, etc.) every dude after the first must be appended to `CasterQueue` as a separate `BuffProvider`. Returning early on key collision (the original `AddSong` shape generalized to `AddActivatable`) leaves the UI grid red/non-interactable for companions because `BuffProvider.CanTarget(unit)` for activatables is `targetId == who.UniqueId` — without a companion provider, the companion target portrait greys out and the user can't mark the toggle wanted on that character. Each provider needs its own per-unit `ActivatableAbility` instance stored on it (`BuffProvider.ActivatableSource`) so the executor can flip `IsOn` on the correct fact, not on the first dude's instance. `ValidateActivatable` and Phase 0 must iterate `ActualCastQueue` per provider for class toggles/abilities/equipment; `Category.Song` keeps the single-performer/mass-effect shape (one bard, listeners are `wanted`).
- **Runtime BubbleBuff state and `SavedState.Buffs[key]` are parallel mirrors**: `wanted`/`notWanted`/`InGroups`/etc. live on both. `RecalculateAvailableBuffs` clears `BuffsByKey`, rebuilds from spellbooks, and calls `InitialiseFromSave(SavedState.Buffs[key])` to rehydrate — so any group change (party swap, Show/Hide Reserve, area transition, level-up) silently restores runtime from SavedState. Bulk-mutation sites (`ClearAllAssignments` and any future "reset/forget-all" feature) must mirror their change into `SavedState.Buffs[*]` and call `Save(true)`, otherwise the runtime change is reverted on the next group event. Single-edit `SetUnitWants` is fine because the deep `Save()` path syncs runtime → SavedState; the trap is bulk sites that skip Save. See v1.13.10 fix in `BufferState.ClearAllAssignments`.
- **`Save()` create/retain conditions must include per-caster config**: `SavedBuffState` entries are created/pruned based on `Requested`/`IgnoreForOverwriteCheck`/`Blacklisted`/groups — caster bans/caps/arcanist flags live only in `save.Casters` and need `hasCasterConfig(buff)` (create, current queue) and `hasSavedCasterConfig(save)` (retain — saved dict also covers providers temporarily absent: depleted scrolls, benched casters). Without both, bans evaporate on the next provider rebuild. See v1.14.5 fix in `BufferState.Save`.
